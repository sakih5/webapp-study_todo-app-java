# プロジェクト企画の思考プロセス - 要件定義の前に考えるべきこと

## 本記事を作成した背景

ソフトウェア開発では「要件定義」が重要だとよく言われますが、その前段階である「企画・構想」フェーズについてはあまり語られません。しかし実際には、「そもそも何を作るべきか？」「本当にそれを作る価値があるのか？」という問いに答えることが、プロジェクトの成否を大きく左右します。

今回、Todoアプリのプロジェクトを例に、企画段階でどのような思考プロセスを経るべきかを言語化しました。個人開発であっても、企業のプロジェクトであっても、この思考プロセスは役立つはずです。

## 本記事で取り組んだこと

- プロジェクト企画の全体像と流れの理解
- 課題・ニーズの特定方法
- ソリューションの妥当性検証
- 実現可能性の評価（技術・予算・期間）
- 競合分析と差別化ポイントの発見
- プロジェクトゴールの設定
- 企画書・プロジェクトチャーターの作成

## 手順

### 前提

- **目的**: Todoアプリを作るべきかどうかを判断する
- **前提知識**: 特になし（企画段階では技術知識よりもビジネス・ユーザー視点が重要）
- **成果物**: 企画書またはプロジェクトチャーター

### 1. 課題・ニーズの特定 - 「誰が何に困っているのか？」

#### 🎯 目的

ソフトウェア開発の出発点は「解決すべき課題」です。課題が明確でなければ、作ったものが誰にも使われない可能性があります。

#### 🛠️ 手順詳細

以下の質問を自分自身（または想定ユーザー）に投げかけます：

**質問1: 誰が困っているのか？（ペルソナの特定）**

```
例：
- 毎日多くのタスクを抱えるビジネスパーソン
- 複数のプロジェクトを並行する学生
- 家事・育児・仕事を両立する主婦/主夫
- フリーランスでスケジュール管理が苦手な人
- 時間の使い方を改善したい自己啓発志向の人
```

**質問2: どんな問題を抱えているのか？**

```
例：
- やるべきことを忘れてしまう
- タスクが多すぎて優先順位がつけられない
- 時間がどこに消えているかわからない
- 計画と実際の時間にズレがある
- 振り返りができず、同じミスを繰り返す
```

**質問3: 現在どう対処しているのか？（現状の解決策）**

```
例：
- 紙のメモやノートに書いている → 検索できない、無くす
- スマホのメモアプリを使っている → 時間管理機能がない
- Googleカレンダーを使っている → タスク管理には不向き
- TrelloやAsanaを使っている → 時間の可視化ができない
- 何も使っていない → 全て頭の中、ストレス
```

**質問4: 既存の解決策の不満点は？（ペインポイント）**

```
例：
- 既存Todoアプリは機能が多すぎて複雑
- 時間の記録と可視化ができるツールが少ない
- 複数ツールを併用しているが、連携が面倒
- 無料版では機能制限が厳しい
- データが企業のサーバーに保存されるのが不安
```

#### 💡 理解ポイント

**なぜ課題特定が最初なのか？**

ソフトウェア開発でよくある失敗パターン：

```
❌ 「Todoアプリを作りたい」から始める
   → 技術的には作れても、誰も使わない

✅ 「時間管理に困っている人がいる」から始める
   → 本当に必要とされるものを作れる
```

**課題の深掘り方法：「5回のなぜ」**

表面的な課題だけでなく、根本原因を探ります。

```
1回目: なぜタスク管理が必要？
  → やるべきことを忘れるから

2回目: なぜ忘れる？
  → タスクが多すぎて頭で管理しきれない

3回目: なぜ頭で管理している？
  → 適切なツールがない / 既存ツールが使いにくい

4回目: なぜ既存ツールが使いにくい？
  → 機能が多すぎて複雑 / 時間管理機能がない

5回目: なぜ時間管理機能が必要？
  → 「何にどれくらい時間を使ったか」を振り返りたい
```

この深掘りによって、「シンプルで時間管理に特化したTodoアプリ」という方向性が見えてきます。

**ペルソナの具体化**

「誰でも使える」ではなく、「こういう人に刺さる」を明確にします。

```
ペルソナ例：
- 名前: 田中太郎（仮）
- 年齢: 28歳
- 職業: Webマーケター
- 状況: 複数のプロジェクトを担当、常にタスクに追われている
- 課題: 時間の使い方が下手で、いつも納期ギリギリ
- 目標: 時間の使い方を可視化して、効率を上げたい
- 現在の行動: Trelloでタスク管理、Togglで時間記録（別々で面倒）
```

#### 📝 補足

**個人開発の場合**

企業のプロジェクトでなく個人開発の場合、「自分が困っている」が出発点でOKです。

```
自分の課題:
- 毎日何にどれくらい時間を使っているか把握したい
- Todoアプリとタイマーアプリを別々に使うのが面倒
- 週末に振り返りをして、来週の改善につなげたい

→ これを解決するTodoアプリを作ろう
```

ただし「自分が困っている = 他の人も困っている」とは限らないので、後で他の人にも需要があるか検証します。

### 2. ソリューションの検討 - 「本当にTodoアプリで解決できるのか？」

#### 🎯 目的

特定した課題に対して、「Todoアプリを作る」が本当に最適な解決策なのかを検証します。

#### 🛠️ 手順詳細

以下の選択肢を比較検討します：

**選択肢1: 既存ツールを使う**

```
メリット:
- 開発コストゼロ
- すぐに使い始められる
- バグが少なく安定している

デメリット:
- 自分の要求に完全にはマッチしない
- 月額課金が必要な場合がある
- カスタマイズできない
```

**選択肢2: 既存ツールを組み合わせる**

```
例: Trello（タスク管理）+ Toggl（時間記録）+ Googleスプレッドシート（分析）

メリット:
- それぞれのツールが高機能
- 開発不要

デメリット:
- ツール間の連携が手間
- データが分散する
- 操作が煩雑
```

**選択肢3: 新規開発する（Todoアプリを作る）**

```
メリット:
- 自分の要求に100%マッチ
- 自由にカスタマイズできる
- 学習になる（個人開発の場合）

デメリット:
- 開発に時間とコストがかかる
- 保守・運用が必要
- 既存ツールより機能が少ない可能性
```

**選択肢4: 業務フローや習慣を変える**

```
例: タスクを減らす、優先順位をつける訓練をする

メリット:
- ツールに頼らず本質的に解決

デメリット:
- 習慣化に時間がかかる
- 根性論になりがち
```

**意思決定マトリクス**

各選択肢を評価します（5段階評価）：

| 選択肢 | コスト | 適合度 | 実現可能性 | 学習価値 | 総合評価 |
|--------|--------|--------|------------|----------|----------|
| 既存ツール使用 | 5 | 2 | 5 | 1 | 13 |
| 既存ツール組み合わせ | 4 | 3 | 5 | 1 | 13 |
| 新規開発 | 2 | 5 | 4 | 5 | 16 |
| 習慣改善 | 5 | 2 | 3 | 2 | 12 |

→ この例では「新規開発」が最も評価が高い

#### 💡 理解ポイント

**「作りたい」と「作るべき」の違い**

```
❌ 「Todoアプリを作りたい」（作ること自体が目的）
   → 完成しても使われない可能性

✅ 「時間管理の課題を解決したい、そのためにTodoアプリが最適」
   → 目的がはっきりしているので、ブレない
```

**新規開発を選ぶべきケース**

1. **既存ツールでは解決できない独自の要求がある**
   - 例: 時間の可視化と振り返り機能が一体化したツールがない

2. **学習目的**
   - 例: Web開発のスキルを身につけたい

3. **ビジネスチャンス**
   - 例: 同じ課題を抱える人が多く、商用化の可能性がある

4. **既存ツールのコストが高すぎる**
   - 例: 月額3,000円のツールを10人で使うと年間36万円

**既存ツールを使うべきケース**

1. **標準的なタスク管理で十分**
   - Todoist、Microsoft To Do、Google Tasksなどで解決できる

2. **すぐに使い始めたい**
   - 開発期間を待てない

3. **開発スキル・リソースがない**
   - 外注コストが高すぎる

#### 📝 補足

**プロトタイプで検証する**

もし判断に迷う場合は、簡易版（プロトタイプ）を作って検証します。

```
1週間で最小限の機能を実装
  ↓
自分や周囲の人に使ってもらう
  ↓
「これなら既存ツールの方が良い」 or 「これは続ける価値がある」
  ↓
判断
```

### 3. 競合分析 - 「既存のTodoアプリと何が違うのか？」

#### 🎯 目的

既存の競合ツールを分析し、差別化ポイント（独自の価値）を見つけます。

#### 🛠️ 手順詳細

**ステップ1: 競合ツールをリストアップ**

主要なTodoアプリを調査します：

```
無料/フリーミアム:
- Todoist
- Microsoft To Do
- Google Tasks
- Notion
- Trello

有料:
- Asana
- ClickUp
- Things 3（Mac/iOS）

時間管理系:
- Toggl Track
- RescueTime
- Clockify
```

**ステップ2: 機能比較表を作成**

| 機能 | Todoist | Notion | Toggl | 自分のアプリ（案） |
|------|---------|--------|-------|-------------------|
| タスク管理 | ◎ | ◎ | - | ◎ |
| 期限日設定 | ◎ | ◎ | - | ◎ |
| 優先度 | ◎ | △ | - | ◎ |
| カテゴリー分類 | ◎（プロジェクト） | ◎ | ◎（プロジェクト） | ◎ |
| サブタスク | ◎ | ◎ | - | ◎（Phase 3） |
| タグ | ◎ | ◎ | ◎ | ◎（Phase 3） |
| 時間記録 | - | - | ◎（タイマー） | ◎（手動入力） |
| 時間可視化 | - | - | ◎（レポート） | ◎（24時間グラフ） |
| 振り返り機能 | △（完了率のみ） | △（手動） | △（週報） | ◎（詳細な統計） |
| シンプルさ | ○ | △（複雑） | ○ | ◎（シンプル重視） |
| 価格 | 無料〜 | 無料〜 | 無料〜 | 無料（オープンソース） |

**ステップ3: 差別化ポイントを特定**

自分のアプリの強みを明確にします：

```
差別化ポイント:
1. タスク管理と時間記録が一体化
   → 他ツール: 別々のアプリを使う必要がある

2. 24時間の可視化（積み上げグラフ）
   → 他ツール: 時間記録はあっても、1日の過ごし方の可視化は弱い

3. 振り返り重視
   → 他ツール: タスクを「こなす」ことに焦点、振り返りは弱い

4. シンプルなUI
   → Notion: 多機能だが複雑
   → Asana: 高機能だがビジネス向けで個人には重い

5. 予定時間と実績時間の比較
   → 他ツール: 計画と実績のギャップを可視化する機能が少ない
```

**ステップ4: ポジショニングマップ**

競合との位置関係を視覚化します：

```
      高機能
        ↑
        │  Asana
        │  ClickUp
        │      Notion
複雑 ←─┼─────────→ シンプル
        │   Todoist
        │   自分のアプリ ★
        │       Google Tasks
        ↓
      シンプル


      時間管理特化
        ↑
        │  Toggl
        │  RescueTime
        │
        │  自分のアプリ ★
        │
        ├─────────→
        │  Todoist
        │  Notion
        │  Trello
        ↓
      タスク管理特化
```

#### 💡 理解ポイント

**「全てに勝つ」を目指さない**

```
❌ 「全ての機能で既存ツールを上回る」
   → 不可能。既存ツールは何年も開発されている

✅ 「特定の領域で圧倒的な価値を提供する」
   → ニッチでも良いので、誰かにとっての"最高のツール"を目指す
```

**差別化の3パターン**

1. **機能差別化**: 他にない機能を提供
   - 例: 24時間の積み上げグラフ

2. **UX差別化**: 同じ機能でも使いやすさで勝つ
   - 例: 3クリックで時間記録完了

3. **価格差別化**: 無料または低価格で提供
   - 例: オープンソース化

**競合分析のポイント**

- 実際に使ってみる（机上の調査だけでは不十分）
- ユーザーレビューを読む（不満点が差別化のヒント）
- 競合の強みは認める（全てで勝つ必要はない）

#### 📝 補足

**競合が多い = チャンスでもある**

```
「Todoアプリは既にたくさんあるから作る意味がない」
         ↓
「既存ツールに不満を持つユーザーがたくさんいる証拠」
         ↓
「差別化できれば、需要は十分ある」
```

例えば、検索エンジン市場はGoogleが支配的でしたが、DuckDuckGoは「プライバシー重視」で差別化し、一定のシェアを獲得しました。

### 4. 実現可能性の評価 - 「本当に作れるのか？」

#### 🎯 目的

「作りたい」「作るべき」と思っても、技術的・時間的・予算的に実現できなければ意味がありません。実現可能性を客観的に評価します。

#### 🛠️ 手順詳細

**評価軸1: 技術的実現可能性**

必要な技術スキルと、現在の自分のスキルレベルを比較します。

```
必要なスキル（スキルマップ）:

フロントエンド:
- HTML/CSS ★★☆☆☆（基本レベル）
- JavaScript ★★★☆☆（DOM操作、fetch API）

バックエンド:
- Python ★★★☆☆（基本文法、関数、クラス）
- FastAPI ★★★★☆（ルーティング、認証、DB連携）

データベース:
- PostgreSQL ★★★☆☆（SQL基本、テーブル設計）
- SQLAlchemy ★★★☆☆（ORM基本）

その他:
- Git ★★☆☆☆（基本コマンド）
- JWT認証 ★★★☆☆（実装経験）
```

**現在のスキルレベルとのギャップ**:

```
自分のスキル:
- HTML/CSS: ★★☆☆☆ → OK
- JavaScript: ★★☆☆☆ → 学習が必要（ギャップ: ★☆☆☆☆）
- Python: ★★☆☆☆ → OK
- FastAPI: ★☆☆☆☆ → 学習が必要（ギャップ: ★★★☆☆）
- PostgreSQL: ★☆☆☆☆ → 学習が必要（ギャップ: ★★☆☆☆）

結論: 学習しながら進めれば実現可能
```

**評価軸2: 時間的実現可能性**

各フェーズの工数を見積もります。

```
Phase 1（MVP）:
- 設計: 1週間
- フロントエンド実装: 2週間
- バックエンド実装: 2週間
- テスト・修正: 1週間
合計: 6週間

Phase 2（時間管理機能）:
- 設計: 3日
- 実装: 2週間
- テスト: 1週間
合計: 3週間

Phase 3（高度な機能）:
- 設計: 1週間
- 実装: 3週間
- テスト: 1週間
合計: 5週間

全体: 約14週間（3.5ヶ月）
```

**リソースとの照合**:

```
利用可能時間:
- 平日: 2時間/日 × 5日 = 10時間/週
- 休日: 4時間/日 × 2日 = 8時間/週
合計: 18時間/週

バッファ（予期せぬ問題）: 50%
実質作業時間: 12時間/週

Phase 1完了までの期間:
6週間 × (40時間/12時間) = 20週間（約5ヶ月）

結論: 個人開発としては長いが、実現可能な範囲
```

**評価軸3: 予算的実現可能性**

開発・運用にかかるコストを見積もります。

```
開発コスト:
- PC、開発環境: 既に持っている → 0円
- 学習教材（書籍、Udemy等）: 10,000円

運用コスト（月額）:
- サーバー（VPS、クラウド）: 1,000円〜
- ドメイン: 100円〜
- データベース: 無料枠 or 1,000円〜
合計: 1,000〜2,000円/月

年間: 12,000〜24,000円

結論: 個人で負担可能な範囲
```

**評価軸4: リスク評価**

プロジェクトを阻害する要因を洗い出します。

```
技術的リスク:
- FastAPIの学習が思ったより難しい → 中リスク
  対策: チュートリアルを先に完了する

- データベース設計のミスで後から修正が大変 → 高リスク
  対策: Phase 1でしっかり設計レビュー

スケジュールリスク:
- 本業が忙しくなり、時間が取れない → 中リスク
  対策: Phase単位で区切り、無理のないペース

モチベーションリスク:
- 途中で飽きる、挫折する → 中リスク
  対策: MVPを早く完成させて達成感を得る

技術変化リスク:
- 開発中にFastAPIの大きな変更がある → 低リスク
  対策: 安定版を使う
```

#### 💡 理解ポイント

**「完璧を目指さない」**

```
❌ 「全てのスキルを身につけてから開発を始める」
   → いつまでたっても始められない

✅ 「基礎を学んだら、開発しながら学ぶ」
   → 実践的なスキルが身につく
```

**工数見積もりの鉄則**

1. **楽観的見積もりは1.5〜2倍になる**
   - 「1週間でできる」 → 実際は2週間

2. **学習コストを忘れない**
   - 既知の技術: 見積もり通り
   - 新規の技術: 見積もりの2倍

3. **バッファを必ず入れる**
   - 予期せぬバグ、設計変更、体調不良など

**実現可能性が低い場合の対処**

```
問題: スキルギャップが大きすぎる
  → 対策: スコープを縮小（Phase 1のみ）、または技術スタックを変更

問題: 時間が足りない
  → 対策: 機能を減らす、リリース時期を延ばす、共同開発者を探す

問題: 予算がない
  → 対策: 無料サービスを使う（Heroku無料枠、Vercelなど）
```

#### 📝 補足

**Go/No-Go判断の基準**

以下の条件を満たせば「Go（実行）」：

```
✓ 技術的に実現可能（学習すれば可能も含む）
✓ 時間的に実現可能（長期でも許容できる）
✓ 予算的に実現可能（負担できる範囲）
✓ リスクが管理可能（対策が立てられる）
```

1つでも「No」なら、プロジェクトを見直すか、条件を変更します。

### 5. プロジェクトゴールの設定 - 「何を達成すれば成功なのか？」

#### 🎯 目的

プロジェクトの成功基準を明確にします。これがないと、「いつまでも完成しない」「完成しても満足できない」状態になります。

#### 🛠️ 手順詳細

**SMART原則でゴールを設定**

```
S (Specific)   : 具体的
M (Measurable) : 測定可能
A (Achievable) : 達成可能
R (Relevant)   : 関連性がある
T (Time-bound) : 期限がある
```

**悪いゴール設定の例**:

```
❌ 「良いTodoアプリを作る」
   → 何が「良い」のか不明確
   → 測定できない
   → 期限がない
```

**良いゴール設定の例**:

```
✅ Phase 1のゴール（3ヶ月後）:
「自分が毎日使える、基本的なTodoアプリを完成させる」

具体的な成功基準:
- ユーザー登録・ログインができる
- Todoの作成・編集・削除・完了ができる
- カテゴリー分類ができる
- スマホでも使える（レスポンシブ対応）
- 自分が1週間使い続けられる

✅ Phase 2のゴール（Phase 1完了後2ヶ月以内）:
「時間管理と振り返りができるTodoアプリを完成させる」

具体的な成功基準:
- Todoに予定時間・実績時間を記録できる
- 日別の時間分布が可視化できる
- カテゴリー別の時間集計ができる
- 自分の時間の使い方の改善につながる

✅ Phase 3のゴール（Phase 2完了後2ヶ月以内）:
「他のユーザーにも使ってもらえるTodoアプリを公開する」

具体的な成功基準:
- サブタスク、タグ、繰り返しタスクが使える
- 24時間の積み上げグラフが表示できる
- 週/月の統計が見られる
- 5人の友人に使ってもらい、フィードバックをもらう
- GitHub/個人サイトで公開する
```

**KPI（重要業績評価指標）の設定**

プロジェクトの進捗と成果を測る指標を決めます。

```
開発プロセスのKPI:
- コミット頻度: 週3回以上
- Phase 1完了: 3ヶ月以内
- バグ修正率: 発見から1週間以内に90%修正

プロダクトのKPI:
- 自分の利用継続率: 30日間で90%以上
- タスク完了率: 週に80%以上
- 時間記録率: 完了したタスクの50%以上に時間記録

ユーザー獲得のKPI（Phase 3以降）:
- 月間アクティブユーザー: 10人
- ユーザー満足度: 5段階評価で平均4以上
- バグ報告率: ユーザー10人あたり月1件以下
```

**中間マイルストーン**

大きなゴールを小さなマイルストーンに分割します。

```
Week 1-2: 技術学習
  → FastAPIチュートリアル完了
  → PostgreSQL基本操作習得

Week 3-4: 設計
  → データモデル設計完了
  → API設計完了
  → 画面設計完了

Week 5-8: バックエンド実装
  → 認証API実装
  → Todo CRUD API実装

Week 9-12: フロントエンド実装
  → ログイン画面
  → Todo一覧・作成画面

Week 13-14: テスト・デバッグ
  → 自分で1週間使ってみる
  → バグ修正
```

#### 💡 理解ポイント

**なぜゴール設定が重要か？**

```
ゴールがない場合:
「なんとなく作り始める」
  ↓
「機能を追加し続ける」
  ↓
「いつまでも完成しない」
  ↓
「途中で飽きる・挫折する」

ゴールがある場合:
「Phase 1を3ヶ月で完成させる」
  ↓
「スコープを守る」
  ↓
「3ヶ月後にMVP完成」
  ↓
「達成感 → 次のPhaseへ」
```

**ゴール設定の落とし穴**

```
❌ 高すぎるゴール:
「3ヶ月でAsanaを超えるアプリを作る」
  → 達成不可能、挫折する

❌ 低すぎるゴール:
「1年かけて画面を1つ作る」
  → モチベーション維持が難しい

✅ 適切なゴール:
「3ヶ月で自分が使える基本機能を完成」
  → 達成可能で、やりがいがある
```

**ゴールは変更してもいい**

プロジェクトを進める中で、ゴールを見直すことは正常です。

```
当初のゴール: 3ヶ月でPhase 1完成
  ↓
実際: 2ヶ月経過時点で50%しか進んでいない
  ↓
見直し: ゴールを4ヶ月に延長 or 機能を削減
  ↓
現実的なゴールに調整
```

#### 📝 補足

**個人開発のゴール設定**

企業プロジェクトと異なり、個人開発では「学習」もゴールになります。

```
技術的ゴール:
- FastAPIを使えるようになる
- データベース設計の経験を積む
- 認証の実装方法を学ぶ

キャリア的ゴール:
- ポートフォリオに載せられる作品を作る
- GitHubで公開してスター獲得
- 技術ブログ記事を書く
```

### 6. プロジェクトチャーター（企画書）の作成 - 全てを1つの文書に

#### 🎯 目的

これまでの検討内容を1つの文書にまとめます。これが「プロジェクトの憲法」となり、迷ったときに立ち戻る基準になります。

#### 🛠️ 手順詳細

プロジェクトチャーターの構成：

```markdown
# Todoアプリ開発 プロジェクトチャーター

## 1. プロジェクト概要

### プロジェクト名
時間管理特化型Todoアプリ「TimeFlow」（仮称）

### プロジェクトの背景
多くの人がタスク管理に困っているが、既存のTodoアプリは「タスクをこなすこと」に焦点があり、「時間の使い方を振り返る」機能が弱い。タスク管理と時間記録を別々のツールで行うのは手間がかかる。

### プロジェクトの目的
- タスク管理と時間記録を一体化したシンプルなTodoアプリを開発する
- 1日の過ごし方を可視化し、振り返りを通じて時間の使い方を改善する
- Web開発のスキルを実践的に習得する（学習目的）

## 2. 課題とソリューション

### 解決する課題
- ペルソナ: 28歳のWebマーケター、複数プロジェクトを担当
- 課題: 時間の使い方が下手で、いつも納期ギリギリ
- ペインポイント:
  - タスク管理ツール（Trello）と時間記録ツール（Toggl）を別々に使うのが面倒
  - 1日の時間の使い方が可視化されない
  - 振り返りができず、同じ失敗を繰り返す

### ソリューション
- タスク管理と時間記録を1つのアプリで完結
- 24時間の積み上げグラフで1日の過ごし方を可視化
- カテゴリー別・期間別の統計で振り返りをサポート

## 3. 差別化ポイント

| 機能 | Todoist | Toggl | 当アプリ |
|------|---------|-------|----------|
| タスク管理 | ◎ | - | ◎ |
| 時間記録 | - | ◎ | ◎ |
| 時間可視化 | - | ○ | ◎（24時間グラフ） |
| シンプルさ | ○ | ○ | ◎ |

競合優位性:
- タスク管理と時間記録の一体化
- 1日の過ごし方の可視化（24時間積み上げグラフ）
- シンプルで使いやすいUI

## 4. スコープ

### Phase 1（MVP） - 3ヶ月
- ユーザー認証
- 基本的なTodo管理（CRUD）
- カテゴリー分類
- レスポンシブUI

### Phase 2 - 5ヶ月
- 時間記録（予定時間・実績時間）
- フィルタリング・ソート・検索
- 日別・カテゴリー別の時間可視化

### Phase 3 - 7ヶ月
- サブタスク、タグ、繰り返しタスク
- 24時間積み上げグラフ
- 週/月単位の統計

### スコープ外（将来的な拡張案）
- モバイルアプリ
- チーム機能
- 外部サービス連携

## 5. 技術スタック

- フロントエンド: HTML, JavaScript
- バックエンド: Spring Boot (Java)
- データベース: PostgreSQL
- 認証: JWT (Spring Security)
- ホスティング: Heroku / AWS（検討中）

## 6. 実現可能性

### 技術的実現可能性
- Spring Boot, PostgreSQLの学習が必要（ギャップ: 中）
- 学習しながら進めれば実現可能

### 時間的実現可能性
- Phase 1: 3ヶ月（週12時間作業）
- 全体: 7ヶ月

### 予算的実現可能性
- 開発コスト: 10,000円（学習教材）
- 運用コスト: 1,000〜2,000円/月
- 個人で負担可能

### リスク
- Spring Boot学習の難易度（中リスク）
- 本業の多忙によるスケジュール遅延（中リスク）
- モチベーション低下（中リスク）

## 7. プロジェクトゴール

### Phase 1のゴール
「自分が毎日使える基本的なTodoアプリを3ヶ月で完成させる」

成功基準:
- ユーザー登録・ログインができる
- Todoの作成・編集・削除・完了ができる
- カテゴリー分類ができる
- 自分が1週間継続して使える

### 最終ゴール
「他のユーザーにも使ってもらえるTodoアプリを7ヶ月で完成・公開する」

成功基準:
- Phase 3の全機能が実装されている
- 5人の友人が使ってくれる
- GitHubで公開し、READMEを整備
- 技術ブログ記事を1本書く

## 8. KPI

開発プロセス:
- コミット頻度: 週3回以上
- Phase完了時期: 計画通り±2週間以内

プロダクト:
- 自分の利用継続率: 30日間で90%以上
- バグ修正率: 発見から1週間以内に90%

## 9. プロジェクト体制

- 開発者: 自分1人
- レビュアー: 友人エンジニア（任意）
- テストユーザー: 友人5人（Phase 3）

## 10. 承認

このプロジェクトチャーターを承認し、開発を開始する。

承認日: 2025-12-14
```

#### 💡 理解ポイント

**プロジェクトチャーターの役割**

1. **方向性の明確化**
   - 「何を作るか」「なぜ作るか」が一目でわかる

2. **スコープクリープの防止**
   - 「これは入れたい」という誘惑に対して、「チャーターのスコープ外」と判断できる

3. **ステークホルダーとの合意**
   - 企業プロジェクトなら上司・クライアントと合意
   - 個人開発なら自分自身との契約

4. **プロジェクトの振り返り**
   - 完了後に「当初の目的を達成できたか？」を評価

**スコープクリープとは？**

```
スコープクリープ（Scope Creep）:
プロジェクト途中で要求が膨らみ続ける現象

例:
当初: 「シンプルなTodoアプリ」
  ↓
開発中: 「カレンダー連携も欲しい」
  ↓
さらに: 「AI でタスクを提案してほしい」
  ↓
結果: スコープが膨大になり、完成しない

対策: チャーターに立ち戻り、「Phase 3以降」と明確化
```

**チャーターは変更できる**

プロジェクトチャーターは「一度決めたら絶対」ではありません。

```
変更が必要なケース:
- 技術的に実現不可能な要求があった
- 市場環境が変わった（競合が同じ機能をリリース）
- リソース（時間・予算）が大幅に変わった

変更プロセス:
1. 変更理由を明確にする
2. 影響範囲を評価する（スコープ、スケジュール、予算）
3. チャーターを更新する
4. 変更履歴を記録する
```

#### 📝 補足

**企業プロジェクトでの違い**

個人開発とは異なり、企業プロジェクトでは以下が追加されます：

```
- 予算計画（詳細な見積もり）
- 人員計画（誰が何をするか）
- リスク管理計画（詳細なリスク対策）
- ステークホルダー分析（誰に影響するか）
- ROI分析（投資対効果）
```

個人開発では、これらを簡略化しても問題ありません。

### 7. Go/No-Go判断 - 「本当にやるのか、やめるのか」

#### 🎯 目的

これまでの分析を総合して、プロジェクトを実行するか（Go）、中止するか（No-Go）を最終判断します。

#### 🛠️ 手順詳細

**判断基準のチェックリスト**

```
□ 解決すべき明確な課題がある
□ 既存ツールでは解決できない（またはより良い解決策を提供できる）
□ 差別化ポイントが明確
□ 技術的に実現可能（学習すれば可能も含む）
□ 時間的に実現可能（期限内に完成できる）
□ 予算的に実現可能（負担できる範囲）
□ リスクが管理可能
□ 明確なゴールがある
□ 自分がやる理由がある（情熱、学習、ビジネスなど）
```

**判断マトリクス**

| 項目 | 評価 | 理由 |
|------|------|------|
| 課題の明確性 | ◎ | 時間管理の課題は実在する |
| ソリューションの妥当性 | ○ | 既存ツールでも可能だが、一体化に価値がある |
| 差別化 | ○ | 時間可視化は強み、ただし競合も多い |
| 技術的実現可能性 | ○ | 学習が必要だが可能 |
| 時間的実現可能性 | ○ | 7ヶ月は長いが許容範囲 |
| 予算的実現可能性 | ◎ | 月2,000円は問題なし |
| リスク | ○ | 中リスク、対策可能 |
| 情熱・モチベーション | ◎ | 自分も困っている、作りたい |

評価:
- ◎: 3個
- ○: 5個
- △: 0個
- ×: 0個

**総合判断: Go（実行）**

理由:
- 致命的な問題（×）がない
- 自分自身が困っている課題を解決できる
- 学習目的としても価値がある
- リスクは管理可能
- モチベーションが高い

**No-Goになるケース（例）**

```
ケース1: 技術的に実現不可能
評価: 「AI による高度な時間予測」が必須要件
理由: 機械学習の知識がなく、学習に1年以上かかる
判断: No-Go（スコープを変更するか、要件から外す）

ケース2: 予算が合わない
評価: 運用コストが月10万円かかる
理由: 個人で負担できない
判断: No-Go（無料サービスに変更するか、中止）

ケース3: 差別化できない
評価: 既存ツール（Todoist）で十分
理由: 独自の価値提供ができない
判断: No-Go（既存ツールを使う）
```

**条件付きGoの場合**

完全にGoでもNo-Goでもない場合、条件を設定します。

```
条件付きGo: Phase 1のみ実行し、その後判断

条件:
- Phase 1完了後、自分が1ヶ月使い続けられたらPhase 2へ
- 使い続けられなかった場合、プロジェクト中止

理由:
- 全フェーズのコミットは重すぎる
- MVPで価値を検証してから判断したい
```

#### 💡 理解ポイント

**「やりたい」と「やるべき」と「やれる」**

```
やりたい: 自分の情熱
やるべき: 課題解決の必要性
やれる: 実現可能性

3つ全て揃う → Go
1つでも欠ける → 要検討
```

**サンクコスト（埋没費用）に注意**

```
❌ 「既に10時間調査したから、やるしかない」
   → サンクコスト（過去の投資）に囚われている

✅ 「今の情報で判断すると、やらない方が良い」
   → 過去は関係なく、今後の価値で判断
```

**No-Goは失敗ではない**

```
No-Goの判断 = 成功

理由:
- 無駄な開発に時間を使わずに済んだ
- リソースを別のプロジェクトに使える
- 企画段階で問題を発見できた（開発後より早い）
```

#### 📝 補足

**個人開発では「やりたい」も重要**

企業プロジェクトと異なり、個人開発では「情熱」も判断基準です。

```
企業プロジェクト:
- ROI（投資対効果）が最重要
- 「やりたい」だけでは不十分

個人開発:
- 学習目的なら「やりたい」も十分な理由
- ただし「完成させる」コミットメントは必要
```

**ピボット（方向転換）の選択肢**

No-Goと判断しても、プロジェクトを完全に諦める必要はありません。

```
元の企画: フル機能のTodoアプリ
  ↓
ピボット案1: 時間可視化機能だけに特化したツール
ピボット案2: 既存Todoアプリの拡張機能（ブラウザ拡張など）
ピボット案3: 学習目的で簡易版を作る（公開はしない）
```

## 学び・次に活かせる知見

### 企画段階の鉄則

1. **課題ファーストで考える**
   - 「何を作りたいか」ではなく「何を解決したいか」から始める
   - 課題が明確でないプロジェクトは失敗しやすい

2. **既存ソリューションを徹底的に調査する**
   - 車輪の再発明を避ける
   - 既存ツールで解決できるなら、それを使う
   - 本当に差別化できるか冷静に判断

3. **実現可能性を楽観視しない**
   - 工数見積もりは1.5〜2倍になる前提で
   - 技術的難易度を甘く見ない
   - 「やりながら学ぶ」にも限界がある

4. **ゴールを明確にする**
   - SMARTの原則（具体的、測定可能、達成可能、関連性、期限）
   - ゴールがないプロジェクトは永遠に完成しない

5. **Go/No-Go判断を恐れない**
   - No-Goは失敗ではなく、成功
   - サンクコストに囚われない
   - 条件付きGo（Phase 1のみ）も有効

### 課題発見の方法

1. **自分の困りごとから始める**
   - 最も確実な課題の発見方法
   - ただし「自分 = 全員」ではないことを忘れずに

2. **「5回のなぜ」で根本原因を探る**
   - 表面的な課題ではなく、本質的な課題を見つける
   - 深掘りすることで、真の価値提案が見えてくる

3. **ペインポイントを具体化する**
   - 「不便」だけでなく、「どう不便」かを明確に
   - 数値化できると強い（「1日3回切り替えが面倒」など）

### 競合分析のコツ

1. **実際に使ってみる**
   - 机上の調査だけでは不十分
   - ユーザー視点で体験することが重要

2. **レビューを読む**
   - ユーザーの不満点が差別化のヒント
   - 高評価だけでなく、低評価も重要

3. **「全てに勝つ」を目指さない**
   - ニッチでも特定の領域で圧倒的な価値を提供
   - 「誰か」にとっての最高のツールを目指す

### 実現可能性評価の注意点

1. **スキルギャップを正直に評価する**
   - 「なんとかなる」ではなく、「学習時間を含めて見積もる」
   - 新しい技術は見積もりの2倍かかる前提で

2. **バッファは必須**
   - 予期せぬ問題は必ず起きる
   - 最低でも50%のバッファを見込む

3. **リスクは対策とセットで考える**
   - リスクを洗い出すだけでは不十分
   - 各リスクに対する対策を必ず立てる

### プロジェクトチャーターの価値

1. **「憲法」として機能する**
   - 迷ったときに立ち戻る基準
   - スコープクリープの防止

2. **変更は悪ではない**
   - 状況が変われば、チャーターも変える
   - ただし変更理由と影響範囲は記録する

3. **個人開発でも作る価値がある**
   - 自分自身との契約
   - モチベーション維持に役立つ

### 企画段階でよくある失敗

1. **「作りたい」が先行する**
   - 課題が後付けになる
   - 誰も使わないものができる

2. **競合調査が甘い**
   - 「こんな便利なものはない」→ 実は既にある
   - 開発後に競合を知って愕然とする

3. **実現可能性を楽観視する**
   - 「1ヶ月でできる」→ 実際は3ヶ月
   - 挫折の原因になる

4. **ゴールが曖昧**
   - 「いいものを作る」では不十分
   - いつまでも完成しない

5. **No-Go判断ができない**
   - サンクコストに囚われて続ける
   - リソースの無駄遣い
